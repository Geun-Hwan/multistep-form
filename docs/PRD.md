# FE-A 다단계 수강 신청 폼 PRD

## 1. 프로젝트 개요

사용자가 단계별로 수강 신청 정보를 입력하고 최종 제출할 수 있는 다단계 폼을 구현한다.

본 문서는 제품 관점에서 필요한 사용자 흐름, 기능 요구사항, 예외 처리, 완료 기준을 정의한다. 구체적인 기술 스택과 구현 구조는 `docs/TECHNICAL_DESIGN.md`에서 다룬다.

---

## 2. 구현 목표

- 사용자는 여러 단계에 걸쳐 수강 신청 정보를 입력할 수 있다.
- 각 단계의 필수값을 입력해야 다음 단계로 이동할 수 있다.
- 이전 단계로 돌아가도 입력한 데이터는 유지된다.
- 특정 선택값에 따라 추가 입력 필드가 노출된다.
- 조건부 필드가 비활성화되면 해당 값은 초기화된다.
- 최종 제출 전 전체 입력값을 다시 검증한다.
- Mock API를 통해 강의 목록 조회와 신청 제출 흐름을 구현한다.
- 모바일 화면에서도 주요 기능을 불편 없이 사용할 수 있도록 한다.

---

## 3. 구현 전제

- 본 프로젝트는 TypeScript 기반의 프론트엔드 과제로 구현한다.
- Mock API 환경은 지원자가 직접 구성한다.
- 실제 서버 연동 없이 강의 목록 조회와 신청 제출 흐름을 확인할 수 있어야 한다.
- 구체적인 기술 스택, 아키텍처, 폴더 구조, 테스트 전략은 `docs/TECHNICAL_DESIGN.md`에서 정의한다.

---

## 4. 사용자 흐름

1. 사용자는 수강 신청 페이지에 진입한다.
2. Step 1에서 기본 정보를 입력한다.
3. Step 1 검증을 통과하면 Step 2로 이동한다.
4. Step 2에서 수강 정보를 입력한다.
5. 선택값에 따라 조건부 입력 필드가 노출된다.
6. Step 2 검증을 통과하면 Step 3으로 이동한다.
7. Step 3에서 입력한 내용을 최종 확인한다.
8. 제출 버튼 클릭 시 전체 데이터를 다시 검증한다.
9. 제출 성공 시 완료 화면을 보여준다.
10. 제출 실패 시 에러 메시지를 보여주고 재시도할 수 있게 한다.

---

## 5. 스텝 구성

### Step 1. 기본 정보 입력

신청자의 기본 정보를 입력받는 단계이다.

| 필드 | 필수 여부 | 검증 조건 |
|---|---|---|
| 이름 | 필수 | 2자 이상 |
| 이메일 | 필수 | 이메일 형식 |
| 전화번호 | 필수 | 숫자만 허용, 10~11자리 |
| 나이대 | 필수 | 선택 필수 |

#### 이동 조건

- 모든 필수값이 유효해야 다음 단계로 이동할 수 있다.
- 유효하지 않은 값이 있으면 각 필드 하단에 에러 메시지를 표시한다.

---

### Step 2. 수강 정보 입력

신청할 강의와 수강 목적을 입력받는 단계이다.

| 필드 | 필수 여부 | 검증 조건 |
|---|---|---|
| 희망 강의 | 필수 | 강의 목록 중 선택 |
| 수강 목적 | 필수 | 선택 필수 |
| 기타 수강 목적 | 조건부 필수 | 수강 목적이 기타일 때 필수 |
| 수강 경험 | 필수 | 선택 필수 |
| 기존 수강 강의명 | 조건부 필수 | 수강 경험이 있음일 때 필수 |

#### 조건부 필드 정책

- 수강 목적이 기타일 경우 기타 수강 목적 입력 필드를 노출한다.
- 수강 목적이 기타가 아닌 값으로 변경되면 기타 수강 목적 값은 초기화한다.
- 수강 경험이 있음일 경우 기존 수강 강의명 입력 필드를 노출한다.
- 수강 경험이 없음으로 변경되면 기존 수강 강의명 값은 초기화한다.

---

### Step 3. 신청 내용 확인

사용자가 입력한 전체 내용을 확인하는 단계이다.

#### 표시 항목

- 이름
- 이메일
- 전화번호
- 나이대
- 희망 강의
- 수강 목적
- 기타 수강 목적, 해당하는 경우만 표시
- 수강 경험
- 기존 수강 강의명, 해당하는 경우만 표시

#### 제출 조건

- 최종 제출 전 전체 입력값을 다시 검증한다.
- 검증 실패 시 문제가 있는 스텝으로 이동한다.
- 제출 중에는 버튼을 비활성화하여 중복 제출을 방지한다.

---

### Step 4. 신청 완료

제출 성공 시 신청 완료 화면을 표시한다.

#### 표시 항목

- 신청 완료 메시지
- 신청 번호
- 신청 일시
- 처음으로 돌아가기 버튼

---

## 6. 스텝 이동 정책

- 다음 단계 이동은 현재 스텝의 유효성 검증을 통과해야 가능하다.
- 이전 단계 이동은 검증 없이 가능하다.
- 이전 단계로 이동해도 입력값은 유지된다.
- 완료된 이전 스텝은 Step Indicator를 통해 이동할 수 있다.
- 아직 도달하지 않은 스텝은 직접 이동할 수 없다.

---

## 7. 브라우저 뒤로가기 정책

스텝 이동은 URL query parameter와 동기화한다.

| 단계 | URL |
|---|---|
| Step 1 | `/course-apply?step=1` |
| Step 2 | `/course-apply?step=2` |
| Step 3 | `/course-apply?step=3` |
| 완료 | `/course-apply?step=complete` |

### 정책

- 브라우저 뒤로가기를 누르면 이전 스텝으로 이동한다.
- 브라우저 앞으로가기를 누르면 접근 가능한 다음 스텝으로 이동한다.
- 입력이 완료되지 않은 스텝으로 직접 접근할 경우 유효한 스텝으로 이동시킨다.
- 잘못된 step 값이 들어오면 Step 1로 이동한다.

---

## 8. 데이터 흐름 요구사항

- 스텝 이동 시 입력값은 초기화되지 않는다.
- 조건부 필드가 숨겨질 경우 해당 값은 초기화한다.
- 숨겨진 조건부 필드 값은 최종 제출 데이터에 포함하지 않는다.
- 최종 제출 시 화면 입력값을 API 요청 형식에 맞게 변환한 뒤 제출한다.

---

## 9. Mock API 요구사항

### 9.1 강의 목록 조회

| 항목 | 내용 |
|---|---|
| Method | GET |
| Endpoint | `/api/courses` |

#### 사용 목적

- Step 2에서 선택 가능한 강의 목록을 조회한다.
- 마감된 강의는 마감 상태로 표시한다.

#### 응답 필드

| 필드 | 타입 | 설명 |
|---|---|---|
| id | string | 강의 ID |
| title | string | 강의명 |
| description | string | 강의 설명 |
| category | string | 강의 카테고리 |
| capacity | number | 총 정원 |
| remainingSeats | number | 잔여 좌석 수 |

---

### 9.2 수강 신청 제출

| 항목 | 내용 |
|---|---|
| Method | POST |
| Endpoint | `/api/course-applications` |

#### 사용 목적

- 최종 검증이 완료된 수강 신청 데이터를 제출한다.

#### 성공 응답 필드

| 필드 | 타입 | 설명 |
|---|---|---|
| applicationId | string | 신청 번호 |
| submittedAt | string | 신청 일시 |
| message | string | 완료 메시지 |

#### 실패 응답 필드

| 필드 | 타입 | 설명 |
|---|---|---|
| message | string | 에러 메시지 |
| code | string | 에러 코드 |

---

### 9.3 예시 데이터 관리

Mock API에서 사용하는 예시 데이터는 `docs/mock-data` 디렉터리에 별도 JSON 파일로 관리한다.

예시 데이터 파일은 API 응답 구조를 문서화하고, Mock API 구성 시 기준 데이터로 활용한다.

| 파일 | 용도 |
|---|---|
| `docs/mock-data/courses.json` | 강의 목록 조회 응답 예시 |
| `docs/mock-data/application-success.json` | 수강 신청 성공 응답 예시 |
| `docs/mock-data/application-error.json` | 수강 신청 실패 응답 예시 |

---

## 10. 모바일 대응 요구사항

- 최소 360px 너비에서도 화면이 깨지지 않도록 구현한다.
- 입력 필드는 모바일에서 전체 너비를 사용한다.
- 주요 CTA 버튼은 모바일에서 하단 고정 영역에 배치한다.
- 버튼과 입력 요소는 터치하기 쉬운 크기로 구성한다.
- Step Indicator는 모바일에서 간결하게 표시한다.

---

## 11. 예외 처리

| 상황 | 처리 방식 |
|---|---|
| 강의 목록 로딩 중 | 로딩 UI 표시 |
| 강의 목록 조회 실패 | 에러 메시지와 재시도 버튼 표시 |
| 마감된 강의 선택 | 신청 불가 안내 |
| 유효하지 않은 입력 | 필드별 에러 메시지 표시 |
| 제출 중 | 제출 버튼 비활성화 |
| 제출 실패 | 에러 메시지 표시 후 재시도 가능 |
| 마감 강의가 우회 선택되어 제출된 경우 | 서버 응답 에러 메시지를 표시하고 제출을 완료 처리하지 않음 |
| 잘못된 step 접근 | 유효한 스텝으로 이동 |

---

## 12. 완료 기준

- 사용자는 Step 1부터 Step 3까지 순서대로 이동할 수 있다.
- 각 스텝의 필수값 검증이 정상적으로 동작한다.
- 조건부 필드가 선택값에 따라 노출되고 숨겨진다.
- 숨겨진 조건부 필드 값은 초기화된다.
- 이전 단계로 돌아가도 입력값이 유지된다.
- 브라우저 뒤로가기/앞으로가기가 스텝 이동과 자연스럽게 연결된다.
- Mock API를 통해 강의 목록 조회와 신청 제출이 동작한다.
- 모바일 화면에서도 주요 기능을 사용할 수 있다.
- 주요 사용자 흐름이 E2E 테스트로 검증된다.
