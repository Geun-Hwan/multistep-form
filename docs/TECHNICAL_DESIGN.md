# FE-A 기술 설계 문서

## 1. 문서 목적

본 문서는 `docs/PRD.md`에 정의된 다단계 수강 신청 폼 요구사항을 구현하기 위한 기술 스택, 아키텍처, 상태 관리, Mock API, 테스트 전략을 정의한다.

PRD가 “무엇을 만들 것인가”를 다룬다면, 본 문서는 “어떻게 구현할 것인가”를 다룬다.

---

## 2. 기술 스택

| 구분 | 기술 | 사용 이유 |
|---|---|---|
| Language | TypeScript | 폼 데이터, API 응답, 컴포넌트 props 타입을 명확히 관리하기 위해 사용 |
| Framework | Next.js App Router | 페이지 라우팅과 화면 구조를 명확하게 구성하기 위해 사용 |
| UI | React | 스텝별 화면을 컴포넌트 단위로 분리하기 위해 사용 |
| Form | React Hook Form | 다단계 폼 상태 유지와 필드 단위 검증을 효율적으로 처리하기 위해 사용 |
| Validation | Zod | 조건부 필드를 포함한 유효성 검증을 스키마로 관리하기 위해 사용 |
| Server State | TanStack Query | 강의 목록 조회, 제출 상태, 에러 상태를 명확히 분리하기 위해 사용 |
| Mock API | MSW | 실제 서버 없이 API 연동 흐름을 검증하기 위해 사용 |
| Styling | Tailwind CSS | 모바일 대응과 UI 스타일링을 빠르게 구현하기 위해 사용 |
| E2E Test | Playwright | 주요 사용자 흐름을 브라우저 환경에서 검증하기 위해 사용 |

---

## 3. 아키텍처 방향

본 프로젝트는 Feature-based Architecture를 적용한다.

수강 신청 기능은 하나의 독립된 도메인으로 볼 수 있기 때문에, 관련 컴포넌트, 훅, 타입, 스키마, API 로직을 `course-apply` 기능 단위로 묶는다.

### 선택 이유

- 기능 관련 코드가 한 위치에 모여 있어 수정 범위를 파악하기 쉽다.
- 컴포넌트, 폼 검증, API 요청, 유틸 함수를 역할별로 분리할 수 있다.
- 과제 규모에서 과도한 레이어드 구조보다 단순하고 명확하다.
- 추후 다른 신청 폼이나 기능이 추가될 경우 feature 단위로 확장하기 쉽다.

## 4. 폴더 구조

예상 폴더 구조는 다음과 같다.

| 경로 | 역할 |
|---|---|
| `src/app/course-apply` | 수강 신청 페이지 엔트리 |
| `src/features/course-apply/components` | 스텝 화면, 네비게이션, 인디케이터 등 UI 컴포넌트 |
| `src/features/course-apply/hooks` | 스텝 이동, 폼 상태 관련 커스텀 훅 |
| `src/features/course-apply/schemas` | Zod 기반 유효성 검증 스키마 |
| `src/features/course-apply/types` | 폼 값, API 요청/응답 타입 정의 |
| `src/features/course-apply/api` | 강의 조회, 신청 제출 API 함수 |
| `src/features/course-apply/utils` | DTO 변환, 에러 필드 매핑 등 유틸 함수 |
| `src/mocks` | MSW 핸들러와 Mock 서버 설정 |
| `docs/mock-data` | Mock API 응답 예시 JSON 데이터 |
| `e2e` | Playwright E2E 테스트 |

## 5. 상태 관리 설계

상태는 성격에 따라 분리해 관리한다.

| 상태 종류 | 관리 방식 | 설명 |
|---|---|---|
| 폼 입력 상태 | React Hook Form | 이름, 이메일, 강의 선택 등 사용자가 입력한 값 |
| 서버 상태 | TanStack Query | 강의 목록 조회, 신청 제출 상태, 로딩 및 에러 상태 |
| 스텝 상태 | URL query parameter | 현재 사용자가 위치한 단계 |
| 일시적 UI 상태 | React local state | 모달, 일시적 에러 표시 등 필요한 경우에만 사용 |

### 설계 기준

- 폼 입력값은 스텝 이동 시 유지되어야 하므로 하나의 FormProvider 안에서 관리한다.
- 강의 목록과 제출 상태는 서버 상태로 보고 TanStack Query로 관리한다.
- 현재 스텝은 URL과 동기화하여 브라우저 뒤로가기/앞으로가기와 연결한다.
- 잘못된 step 값이나 도달 불가능한 step 접근은 허용 가능한 step으로 즉시 보정한다.

## 6. 폼 검증 설계

유효성 검증은 Zod 스키마를 기준으로 정의한다.

### 검증 방식

- 다음 단계 이동 시 현재 스텝에 필요한 필드만 검증한다.
- 최종 제출 시 전체 폼 데이터를 다시 검증한다.
- 조건부 필드는 활성화된 경우에만 필수값으로 검증한다.
- 조건부 필드가 비활성화되면 값을 초기화한다.

### 조건부 필드 처리

- 수강 목적이 기타가 아니면 기타 수강 목적 값을 초기화한다.
- 수강 경험이 없음이면 기존 수강 강의명 값을 초기화한다.
- 숨겨진 조건부 필드 값은 제출 DTO에 포함하지 않는다.

## 7. API 및 Mocking 설계

API 호출은 실제 서버 대신 MSW로 Mocking한다.

| API | Method | 역할 |
|---|---|---|
| `/api/courses` | GET | 강의 목록 조회 |
| `/api/course-applications` | POST | 수강 신청 제출 |

### Mock 데이터 관리

Mock API 응답 예시는 `docs/mock-data`에 JSON 파일로 관리한다. 또한 개발 중 실패 케이스를 검증할 수 있도록 쿼리 파라미터 기반 Mock 모드를 지원한다.

| 파일 | 역할 |
|---|---|
| `docs/mock-data/courses.json` | 강의 목록 조회 응답 예시 |
| `docs/mock-data/application-success.json` | 수강 신청 성공 응답 예시 |
| `docs/mock-data/application-error.json` | 수강 신청 실패 응답 예시 |

### 설계 이유

- API 응답 구조를 문서와 코드에서 일관되게 관리할 수 있다.
- 실제 서버가 없어도 API 연동 흐름을 검증할 수 있다.
- 성공/실패 케이스를 쿼리 파라미터와 Mock 헤더로 제어하기 쉽다.
- 마감 강의 우회 제출처럼 UI 밖 예외도 Mock 계층에서 검증할 수 있다.

## 8. 라우팅 및 스텝 동기화 설계

현재 스텝은 URL query parameter로 표현한다.

| 스텝 | URL |
|---|---|
| Step 1 | `/course-apply?step=1` |
| Step 2 | `/course-apply?step=2` |
| Step 3 | `/course-apply?step=3` |
| 완료 | `/course-apply?step=complete` |

### 설계 기준

- 다음 단계 이동 시 현재 스텝 검증을 통과한 뒤 URL을 변경한다.
- 이전 단계 이동은 검증 없이 허용한다.
- 브라우저 뒤로가기/앞으로가기는 URL 변경을 기준으로 동작한다.
- 접근할 수 없는 스텝에 직접 접근하면 유효한 스텝으로 보정한다.

## 9. UI 설계 방향

### 모바일 우선 대응

- 최소 360px 너비에서 주요 기능이 깨지지 않도록 구성한다.
- 입력 필드는 모바일에서 전체 너비를 사용한다.
- 주요 CTA는 모바일에서 하단 고정 영역에 배치한다.
- 터치 가능한 버튼과 입력 요소는 충분한 높이를 확보한다.

### 컴포넌트 분리 기준

- 화면 단위 컴포넌트와 공통 UI 컴포넌트를 분리한다.
- 각 Step 컴포넌트는 해당 단계의 필드 렌더링만 담당한다.
- 스텝 이동, 검증, 제출 로직은 컴포넌트 내부에 과도하게 섞지 않고 훅 또는 유틸로 분리한다.

---

## 10. 테스트 전략

주요 사용자 흐름은 Playwright 기반 E2E 테스트로 검증한다.

### 테스트 대상

- 정상적인 수강 신청 완료 흐름
- 필수값 미입력 시 다음 단계 이동 제한
- 조건부 필드 노출 및 값 초기화
- 이전 단계 이동 시 입력값 유지
- 브라우저 뒤로가기 동작
- 제출 성공 화면 표시
- 제출 실패 시 에러 메시지 표시

### 테스트 설계 이유

다단계 폼은 사용자의 입력, 이동, 검증, 제출 흐름이 중요하다. 따라서 개별 함수 테스트보다 실제 브라우저에서의 사용자 흐름 검증이 더 효과적이라고 판단했다.

---

## 11. 제약사항 및 고려사항

- 실제 서버 API는 사용하지 않고 MSW 기반 Mock API로 대체한다.
- 새로고침 후 입력값 복원은 기본 구현 범위에서 제외한다.
- 접근성은 label, 에러 메시지, 버튼 상태 등 기본 요소 중심으로 반영한다.
- 세부 브라우저별 QA는 제한적으로 진행한다.
- 기능 요구사항은 `docs/PRD.md`를 기준으로 하며, 본 문서는 해당 요구사항을 구현하기 위한 기술적 결정만 다룬다.
