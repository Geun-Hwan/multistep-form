# FE-A 다단계 수강 신청 폼

## 프로젝트 개요

사용자가 단계별로 수강 신청 정보를 입력하고 최종 제출할 수 있는 다단계 수강 신청 폼입니다.

본 프로젝트는 스텝 간 데이터 유지, 조건부 필드 처리, 스텝별 유효성 검증, 브라우저 뒤로가기 대응, 모바일 환경 대응, Mock API 연동을 중심으로 구현했습니다.

자세한 기능 요구사항과 기술 설계는 아래 문서를 참고할 수 있습니다.

- 기능 요구사항: `docs/PRD.md`
- 기술 설계: `docs/TECHNICAL_DESIGN.md`
- Mock 데이터: `docs/mock-data`

## 요구사항 해석 및 가정

과제에서 Mock API 구성 방식과 세부 UX 정책은 자유롭게 선택할 수 있었기 때문에, 실제 서비스에 가까운 수강 신청 흐름을 기준으로 요구사항을 해석했습니다.

- 사용자는 여러 단계에 걸쳐 수강 신청 정보를 입력합니다.
- 각 스텝의 필수값이 유효해야 다음 단계로 이동할 수 있습니다.
- 이전 단계로 돌아가도 입력값은 유지됩니다.
- 선택값에 따라 조건부 입력 필드가 노출됩니다.
- 조건부 필드가 비활성화되면 해당 값은 초기화됩니다.
- 강의 목록 조회와 수강 신청 제출은 Mock API로 처리합니다.
- 주요 사용자 흐름은 E2E 테스트로 검증합니다.

## 설계 결정과 이유

### Feature-based Architecture 적용

수강 신청 기능을 하나의 독립된 기능 단위로 보고, 관련 컴포넌트, 훅, 타입, 스키마, API 로직을 `course-apply` 기준으로 묶었습니다.

이 구조는 기능 관련 코드가 한 위치에 모여 있어 수정 범위를 파악하기 쉽고, 과제 규모에서 과도한 구조보다 단순하고 명확하다고 판단했습니다.

### 폼 상태와 서버 상태 분리

사용자가 입력하는 폼 값은 React Hook Form으로 관리하고, 강의 목록 조회와 제출 상태는 TanStack Query로 관리했습니다.

이를 통해 입력 상태와 API 요청 상태를 분리하고, 로딩 및 에러 처리를 명확하게 관리할 수 있도록 했습니다.

### URL 기반 스텝 동기화

브라우저 뒤로가기와 앞으로가기를 자연스럽게 처리하기 위해 현재 스텝을 URL query parameter와 동기화했습니다.

### Mock API 구성

실제 서버 없이도 API 연동 흐름을 확인할 수 있도록 MSW를 사용했습니다. Mock API 응답 예시는 `docs/mock-data`에 JSON 파일로 관리했습니다.

## 미구현 / 제약사항

- 실제 서버 API 연동은 구현하지 않았으며, MSW 기반 Mock API로 대체했습니다.
- 새로고침 후 입력값 복원은 구현하지 않았습니다.
- 파일 업로드, 결제, 로그인 인증 등 수강 신청 외 부가 기능은 구현 범위에서 제외했습니다.
- 강의 목록 데이터는 정적 Mock 데이터를 기준으로 구성했습니다.
- 접근성은 기본 label, 에러 메시지, 버튼 상태 중심으로 반영했습니다.
- 브라우저 및 디바이스별 세부 QA는 제한적으로 진행했습니다.

## AI 활용 범위

본 프로젝트는 AI 도구를 적극 활용하여 개발했습니다.

AI는 다음 작업에 활용했습니다.

- PRD 초안 작성
- 기술 스택 및 라이브러리 구성 정리
- 초기 코드 생성
- 컴포넌트 보일러플레이트 작성
- MSW 기반 Mock API 구성
- E2E 테스트 시나리오 초안 작성
- 코드 개선 방향 검토

저는 구현 방향을 정하고, 생성된 결과물을 직접 실행하며 주요 흐름을 검토했습니다. 특히 스텝 이동, 조건부 필드 처리, 유효성 검증, 모바일 화면, Mock API 응답, E2E 테스트가 의도한 대로 동작하는지 확인하고 필요한 부분을 수정했습니다.
